# Week 1 论文笔记：疑问与技术决策

## 1. 当前阶段定位
第1周的目标不是实现完整 RAG 功能，而是完成项目设置与骨架搭建，建立后续开发所需的工程基础。

## 2. 本周技术决策

### 决策 1：后端框架选用 FastAPI
原因：
- 适合快速搭建 API 服务
- 自带接口文档，便于后续调试和展示
- 与后续 RAG 接口开发衔接自然

### 决策 2：先完成最小骨架，不提前做复杂功能
原因：
- 当前阶段重点是建立可运行、可测试、可扩展的工程基础
- 避免一开始功能过多导致项目失控

### 决策 3：配置使用 `.env` + `config.py`
原因：
- 避免把项目名称、版本号、日志级别等写死在代码中
- 为后续 API Key、模型名、数据目录等配置留入口

### 决策 4：日志先做控制台 + 文件双输出
原因：
- 满足第1周最小排查需求
- 后续导入、切块、检索阶段定位问题更方便

### 决策 5：测试先从 `/health` 开始
原因：
- 先建立最小自动化验证闭环
- 后续可逐步扩展到更多模块

## 3. 本周遇到的问题

### 问题 1：日志模块命名冲突
`logging.py` 与 Python 标准库重名，导致导入错误。

修复：
改名为 `logging_config.py`。

### 问题 2：pytest 导入 `app` 失败
原因：
包结构识别不稳定。

修复：
补充 `__init__.py`，并使用 `python -m pytest` 运行。
